# v15.1a 诊断与 v15.2a Decode-Time Visual Bias 试验报告

## 0. 结论摘要

这一轮实验的核心目的不是继续“堆 planner 小模块”，而是回答两个问题：

1. `v15.1a` 的失败到底来自 inference 时 adapter 使用错误，还是训练已经破坏了主干生成分布？
2. 如果 planner 内部确实有 visual-state 信息，能否通过 frozen/near-frozen 的 decode-time 控制直接改善 `LSE-D / LSE-C`？

目前结论非常明确：

- `v15.1a` 应该从主线中移除。`adapter_off` 和 `zero_planner` 几乎完全相同，说明不是 planner 输入在 inference 的单点问题，而是 `v15.1a` 训练后主干/q0 分布已经被损坏。
- 当前 `v15.1a` 代码本身不是主要问题。把 `v15.0b epoch02` 通过 `v15.1a` 代码、adapter disabled 加载，mini20 可以恢复到接近 `v15.0b` 的水平。
- `v15.2a` 的 q0-token/viseme 统计先验 decode bias 也不是正确方向。最弱权重 `w=0.10` 已经让 system1 的 WER、LSE-D、LSE-C 全部变差，因此我停止了 `w=0.20 / w=0.35` 的继续评估。

## 1. v15.1a 诊断

### 1.1 原始失败

`v15.1a` 的设计是：

$$
g^{(0)}_t \leftarrow g^{(0)}_t + \lambda \Delta(g^{(0)}_t, P_t)
$$

其中：

- $g^{(0)}_t$ 是 q0/codebook0 对应的中间表示。
- $P_t$ 是 planner 在当前视觉时间 $t$ 的状态，包括 `plan_hidden / occurrence / remaining / stop / viseme_logits`。
- adapter gate 初始较小，希望只做轻量修正。

但是 `v15.1a epoch0 mini20` 的外部生成结果非常差，尤其是 system2：

| Model | System | mean WER | LSE-D | LSE-C | UTMOS | SPK |
|---|---:|---:|---:|---:|---:|---:|
| v15.1a epoch0 | system1 | 8.8050 | 10.6188 | 3.4294 | 2.3884 | 0.3560 |
| v15.1a epoch0 | system2 | 16.9811 | 11.3820 | 2.6605 | 2.3164 | 0.2171 |

### 1.2 Adapter-Off 诊断

诊断设置：

- checkpoint: `/data1/jinyu_wang/projects/Glo-Var/artifacts/Glo-Varv15/v1a/epoch00.ckpt`
- config: `glo-var/v15.1a_adapter_off`
- 目的：同一个训练后 checkpoint，inference 时关闭 q0 planner adapter。

结果：

| Model | System | mean WER | LSE-D | LSE-C | UTMOS | SPK |
|---|---:|---:|---:|---:|---:|---:|
| v15.1a adapter-off | system1 | 8.8050 | 10.6188 | 3.4294 | 2.3884 | 0.3560 |
| v15.1a adapter-off | system2 | 16.9811 | 11.3820 | 2.6605 | 2.3164 | 0.2171 |

解释：

- 关闭 adapter 后并没有恢复。
- 因此失败不能归因于 inference 时 adapter 直接扰动 q0。
- 更可能是训练过程中 adapter 与全模型 finetune 一起改变了 decoder / codebook head / routing 的分布，使得即使关闭 adapter，主干也已经偏离。

### 1.3 Zero-Planner 诊断

诊断设置：

- checkpoint: `v15.1a epoch00`
- config: `glo-var/v15.1a_zero_planner`
- 目的：adapter 仍存在，但 planner 输入置零，测试是否 planner signal 本身导致坏输出。

结果：

| Model | System | mean WER | LSE-D | LSE-C | UTMOS | SPK |
|---|---:|---:|---:|---:|---:|---:|
| v15.1a zero-planner | system1 | 8.8050 | 10.6188 | 3.4294 | 2.3884 | 0.3560 |
| v15.1a zero-planner | system2 | 16.9811 | 11.3820 | 2.6605 | 2.3164 | 0.2171 |

解释：

- `zero_planner` 与 `adapter_off` 完全一致。
- planner 输入不是主要变量。
- 这进一步证明 `v15.1a` 的训练结果本身已经失败。

### 1.4 v15.0b Through v15.1a Code

诊断设置：

- checkpoint: `v15.0b epoch02` 的 state_dict，加上 v15.1a adapter init keys 的 bridge ckpt。
- config: `glo-var/v15.1a_adapter_off`
- 目的：确认当前代码路径本身是否破坏老 checkpoint。

结果：

| Model | System | mean WER | LSE-D | LSE-C | UTMOS | SPK |
|---|---:|---:|---:|---:|---:|---:|
| v15.0b e2 through v15.1a code | system1 | 4.4025 | 10.5947 | 3.4484 | 2.4240 | 0.3656 |
| v15.0b e2 through v15.1a code | system2 | 8.1761 | 11.0496 | 3.0424 | 2.2003 | 0.2363 |

解释：

- 当前代码路径可以正常跑旧 checkpoint。
- 所以不是 inference plumbing 的系统性 bug。
- `v15.1a` 的失败应归因于训练设计本身：在 autoregressive generation 极其敏感的 q0 表示上做 train-time residual injection，会把 teacher-forced 的小收益放大成 generation-time 分布漂移。

## 2. v15.2a: Decode-Time Visual Viseme Bias

### 2.1 设计动机

`v15.1a` 的问题是 train-time adapter 伤害主干。因此 `v15.2a` 尝试一个更保守的策略：

- 不新增训练参数。
- 不改变 checkpoint state_dict。
- 使用 frozen `v15.0b epoch02`。
- 只在 decoding 时，对 q0 logits 加一个由 visual-state planner 输出决定的轻量 bias。

设 planner 预测当前时间的 viseme 分布：

$$
p_t(v)=\text{softmax}(z^{viseme}_t)
$$

从训练集统计 q0 token 与 viseme 的共现先验：

$$
B(v, q)=\log p(q \mid v)-\log p(q)
$$

则 decoding 时对 q0 logits 做：

$$
\ell^{(0)}_{t,q}
\leftarrow
\ell^{(0)}_{t,q}

w \sum_v p_t(v) B(v,q)
$$

这里：

- 只作用于 q0。
- 不作用于 q1/q2/q3。
- 不改 transformer hidden state。
- `B(v,q)` 被 clamp，避免极端 token bias。

### 2.2 统计先验构建

脚本：

```bash
python scripts/build_q0_viseme_prior.py \
  --dataset-dir /data1/jinyu_wang/projects/Glo-Var/data/dataset/LRS3_Dataset/mp4/trainval_preprocess \
  --alignment-root /data1/jinyu_wang/projects/Glo-Var/data/dataset/LRS3_Dataset/mp4/trainval_preprocess/mfa_target_alignment \
  --split train \
  --max-samples 50000 \
  --log-every 5000 \
  --num-threads 1 \
  --smoothing 1.0 \
  --clamp 2.5 \
  --output /data1/jinyu_wang/projects/Glo-Var/artifacts/Glo-Varv15/q0_viseme_prior_lrs3_train.pt
```

构建结果：

- used samples: `50,000`
- used frames: `7,269,000`
- missing split / encodec / alignment: `0`
- bad token: `0`
- viseme totals:

```text
[0, 484384, 283114, 882818, 175474, 1417508,
 530394, 512686, 482200, 1588354, 912068]
```

注意：

- `viseme 0 / silence` 的 count 是 `0`。
- 这说明当前 alignment label 主要覆盖有声 phone。
- 因此本轮没有启用 activity/silence bias，否则会引入不可靠的 silence 控制。

- 它甚至在最弱权重 `w=0.10` 下就破坏了 system1。
- 更大的 `w=0.20 / 0.35` 没有继续跑的价值。

## 3. 为什么 v15.2a 失败

### 3.1 q0 token 不是稳定的 viseme 符号

这个实验隐含假设：

$$
q^{(0)}_t \approx \text{visual/phonetic token}
$$

但 EnCodec q0 token 仍然是 acoustic codec token，不是 phoneme/viseme token。它同时包含：

- 音素相关信息；
- speaker/timbre 信息；
- 局部能量；
- 共振峰；
- codec residual 分布；
- 上下文相关的声学状态。

所以 `viseme -> q0 token` 的统计先验非常粗糙。直接把这个先验加到 q0 logits 上，很容易把模型从正确 acoustic manifold 推开。

### 3.2 Planner viseme 自身还不够强

前面 probe 已经显示：

- visual-state probe 能在 raw lip feature 上预测 visual activity / viseme；
- 但模型内部 planner 的 phone/viseme heads 并没有达到足够高的控制精度。

当 $p_t(v)$ 本身不够准时，decode bias 会变成噪声控制。

### 3.3 LSE 不是单帧 token prior 问题

`LSE-D / LSE-C` 反映的是 audio-video temporal alignment。它更像：

$$
\text{local trajectory matching}
$$

而不是：

$$
\text{current-frame token class matching}
$$

也就是说，真正需要控制的是：

- speech onset/offset；
- phoneme transition timing；
- syllable rhythm；
- local progress speed；
- q0 sequence 的整体轨迹。

单帧 q0-token prior 只会局部改变 token distribution，不能保证时序轨迹更同步。

## 4. 当前应该保留和放弃的内容

### 4.1 放弃

- `v15.1a` 作为主线训练版本。
- train-time q0/g0 residual adapter。
- q0-token/viseme 静态统计 prior decode bias。
- 继续强化 `plan_phone / plan_viseme` 的内部 head，希望它自然转化成外部 LSE。

### 4.2 保留

- `v15.0b` 的 planner state 作为可分析模块。
- `v14.6b` / `v14.5` 作为更稳定的外部 baseline。
- q0-dominant delayed-codebook 解释。
- frozen checkpoint 上做 decode-time 控制的实验范式。

### 4.3 下一步方向

下一步不应该再做“当前帧 token prior”。更合理的是做 trajectory-level / candidate-level 的 visual control：

1. **候选级 AV reranking pilot**
   - 用同一个 checkpoint 生成多个候选。
   - 不改模型 logits。
   - 用独立 visual-audio scorer 或内部 planner consistency score 做 rerank。
   - 如果 rerank 能显著改善 LSE，说明模型本身有好候选，只是 sampling 选错。

2. **progress-speed decoding control**
   - 控制的是 q0 轨迹推进速度，而不是具体 token id。
   - 对重复、停滞、过快跳转做 penalty。
   - 目标是降低 WER tail 和 LSE-D。

3. **训练侧只做 safe regularization**
   - 不再直接残差修改 g0/q0。
   - 只约束 hidden-state trajectory 的稳定性或 planner-output consistency。
   - 训练时冻结更多主干，避免再次破坏 generation distribution。

---

## 5. 当前轮结论

这轮实验虽然是负结果，但信息价值很高：

- `v15.1a` 不是一个值得继续训练的分支。
- `v15.2a` 证明“用 planner viseme 静态先验直接推 q0 token”不可行。
- 真正可能有效的方向应该从 local token manipulation 转向 trajectory-level decoding / candidate-level selection。

因此下一轮 auto-review-loop 应该围绕：

> 如何在不破坏强 baseline 的情况下，直接控制生成轨迹的 visual alignment。

而不是继续增加 planner 内部预测头或 q0 residual adapter。
