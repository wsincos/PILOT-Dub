# `PILOT-Dub / Glo-Var v14.5` 详细笔记

## 0. 版本定位 (Version Positioning)

`v14.5` 是当前最适合作为论文 clean base 的生成器版本。

它不是当前最终 reranking 方法本身，而是一个相对干净、结构逻辑清楚的 Visual TTS generator：

```text
v14.5 = v14.4c asymmetric real-time routing
        + delayed-codebook hierarchical residual conditioning
```

它的核心价值在于：

> 在 delayed codebook pattern 下，显式建模同一真实时间上不同 codebook 的 residual 生成关系。

这比 `v15.0 / v15.1` 那种复杂 planner-state controller 更干净，因为它不依赖大量内部 planner head，也不试图直接强行修改 q0/g0 的语义。

---

## 1. 核心目标 (Objective)

当前 Visual TTS / dubbing 模型要解决的问题是：

> 给定目标文本、参考语音、目标视频，生成既听起来自然、又和目标视频唇动同步的语音。

`v14.5` 的具体目标是：

1. 保留 `v14.4c` 中已经相对稳定的 real-time aligned asymmetric routing；
2. 利用 delayed codebook pattern 中天然存在的 codebook 生成顺序；
3. 让 residual codebook `q1/q2/q3` 在预测时显式看到同一真实时间下已经生成的 earlier codebook；
4. 不改变 AR 主体、不额外引入复杂 loss、不引入难解释 planner controller。

一句话：

> `v14.5` 试图把 EnCodec / RVQ 的 residual hierarchy 显式写进 codebook head 的条件结构里。

---

## 2. 核心挑战 (Challenge)

### 2.1 Delayed codebook pattern 让时间关系变复杂

模型不是直接在同一个 step 同时生成真实时间 `t` 的全部 codebook：

```text
q0(t), q1(t), q2(t), q3(t)
```

而是使用 delayed codebook pattern。

直观上，同一真实时间 `t` 的不同 codebook 被分散到不同 AR step：

```text
step t     -> q0(t)
step t + 1 -> q1(t)
step t + 2 -> q2(t)
step t + 3 -> q3(t)
```

因此：

```text
q1(t) 生成时，q0(t) 已经在历史里
q2(t) 生成时，q0(t), q1(t) 已经在历史里
q3(t) 生成时，q0(t), q1(t), q2(t) 已经在历史里
```

这正好符合 residual codec 的层级结构。

### 2.2 q0 和 q1/q2/q3 的角色不对称

在 delayed pattern 下，`q0` 最接近当前真实时间的主内容生成：

```text
q0 = primary / coarse / content-dominant codebook
```

而 `q1/q2/q3` 更像 residual refinement：

```text
q1/q2/q3 = residual acoustic details
```

如果所有 codebook 都只从同一个 `H_t` 独立预测，就没有显式利用这种 residual hierarchy。

### 2.3 不能继续做过强的 q0/g0 干预

后续实验已经证明：

- `v15.1a` train-time q0/g0 adapter 会损坏主干；
- `v15.2a` q0-token prior 会推坏 acoustic token distribution。

这反过来说明 `v14.5` 的价值：

> 它不强行改变 q0，而是只让 residual codebook 使用已有 earlier-codebook 信息。

---

## 3. 核心思想 (Core Idea)

`v14.5` 的核心思想是：

> 对于每个 residual codebook `q > 0`，在预测它时，把同一真实时间下已经生成的 earlier codebook `q' < q` 的 token embedding 汇聚成 residual context，再注入该 codebook 的 hidden state。

也就是说：

```text
q0: 不加 hierarchical residual context
q1: 使用 q0 的 same-real-time context
q2: 使用 q0 + q1 的 same-real-time context
q3: 使用 q0 + q1 + q2 的 same-real-time context
```

这个设计非常重要：

- 它不改 delayed pattern；
- 它不改 AR 生成顺序；
- 它不需要额外标签；
- 它不需要新的 loss；
- 它只把 codec 本身的 residual 结构显式利用起来。

---

## 4. 符号定义 (Notation)

### 4.1 Codebook token

设 EnCodec / RVQ 有 `Q=4` 个 codebook：

```text
q ∈ {0,1,2,3}
```

真实时间轴上的 codec token 记作：

$$
c_{t,q}
$$

其中：

- $t$ 是真实音频时间帧；
- $q$ 是 codebook index。

### 4.2 Delayed step

AR 模型实际运行在 delayed step 上，记作：

$$
s
$$

在直观解释中：

$$
c_{t,q} \text{ 出现在 } s \approx t+q
$$

在代码实现中，由于 pattern sequence 前面存在 blank / special column，`DelayedHierarchicalResidualConditioner` 的注释采用：

$$
t = p - 1 - q
$$

其中：

- $p$ 是 shifted token 序列中的 delayed position；
- `-1` 来自 delayed pattern 的 leading blank 对齐。

二者只是 index convention 不同，语义相同：

> codebook index 越大，同一真实时间的 token 越晚出现在 delayed sequence 中。

### 4.3 Transformer hidden / g / q

为了和前面讨论一致，定义三层：

#### H 层

$$
H_s
$$

表示 AR decoder 在 delayed step `s` 的输出 hidden。

它包含：

- text condition；
- reference audio history；
- target video / planner condition；
- AR historical tokens。

`H_s` 是所有 codebook head 的共同基础。

#### g 层

每个 codebook head 内部会把 `H_s` 或 codebook-specific hidden 投影成中间表示：

$$
g_{s,q}
$$

在代码里对应 `TwoStepVoiceCraftHeads.forward_with_reprs()` 中每个 head 的 representation。

#### q/token 层

最终每个 head 输出 logits：

$$
\ell_{s,q}
$$

采样或 teacher-forcing 目标对应 token：

$$
c_{t,q}
$$

### 4.4 v14.5 的核心变量

对于某个 delayed position `p` 和 codebook `q`：

- 当前要预测的 hidden:

$$
h_{p,q}
$$

- 同一真实时间上 earlier codebook 的 token:

$$
c_{t,q'}, \quad q' < q
$$

- earlier-codebook context:

$$
r_{p,q}
$$

- 注入后的 hidden:

$$
\tilde{h}_{p,q}
$$

## 5. 模型输入 (Model Inputs)

`v14.5` 沿用 `v14.4c / v14.2` 的输入：

- `Text`: reference + target phoneme tokens
- `Audio_ref`: reference speech EnCodec tokens
- `Video_tar`: target lip feature
- `split_lens`: reference / target 分割位置
- `y`: full codec token sequence

shape 近似：

$$
x \in \mathbb{N}^{B \times T_x}
$$

$$
y \in \mathbb{N}^{B \times Q \times T_y}
$$

$$
v \in \mathbb{R}^{B \times T_v \times 1024}
$$

其中：

```text
B = batch size
Q = 4 codebooks
T_x = text length
T_y = audio token length
T_v = video frame length
```

---

## 6. 输入预处理 (Input Preprocessing)

### 6.1 Text embedding

文本 token：

$$
x
$$

经过 embedding：

$$
E_x = \text{TextEmbedding}(x)
$$

再加 position embedding 和 segment embedding：

$$
\tilde{E}_x = \text{TextPosEmb}(E_x) + \text{TextSegmentEmb}
$$

### 6.2 Audio delayed pattern

原始 audio token：

$$
y \in \mathbb{N}^{B \times Q \times T_y}
$$

根据 `split_lens` 分成：

```text
reference segment
target segment
```

然后对每段应用 delayed pattern：

```text
processor.prepare_audio_input()
```

得到：

```text
embedded_y
new_y_lens
rearranged_y
```

其中 `rearranged_y` 保留原始真实时间 target tokens，是后面构造 shifted target token 的依据。

### 6.3 Video / planner

video lip feature：

$$
v \in \mathbb{R}^{B \times T_v \times 1024}
$$

先被上采样：

```text
processor.prepare_video_input()
```

再经过 video tokenizer / projector：

```text
token_out = video_tokenizer(v_upsampled, v_lens)
```

对于 `v14.5` 来说，planner 部分继承自 `v14.4c`：

```text
plan_hidden = plan_encoder(video_embeddings, text_memory, ref_memory, ...)
```

## 7. v14.5 相比 v14.4c 的改动

### 7.1 v14.4c 已有内容

`v14.4c` 的核心是 real-time aligned asymmetric routing：

```yaml
use_codebook_routing: true
use_per_codebook_real_time_routing: true
routing_codebook_scales: [0.05, 0.03, 0.02, 0.02]
```

其作用是：

> 对不同 codebook 按照它们对应的真实时间，选择更合适的 planner/video condition。

也就是修正：

```text
delayed step s
codebook q
real time t = s - q
```

之间的错位。

### 7.2 v14.5 新增内容

`v14.5` 新增：

```yaml
use_hierarchical_codebook_conditioning: true

hierarchical_context_hidden_dim: 512
hierarchical_context_dropout: 0.1
hierarchical_context_scales: [0.0, 0.03, 0.03, 0.03]
```

对应模块：

```text
DelayedHierarchicalResidualConditioner
```

关键点：

```text
q0 scale = 0.0
q1 scale = 0.03
q2 scale = 0.03
q3 scale = 0.03
```

也就是说：

- q0 不接收 earlier-codebook residual context；
- q1/q2/q3 接收；
- residual context 初始强度很小，避免破坏原模型。

## 8. v14.5 训练时完整数据流

### 8.1 Decoder 输出 target hidden

模型先正常经过 decoder：

```text
xy_input
    -> transformer decoder
    -> y_out
```

然后取 target audio 部分：

```python
y_out_target = gather_sequence_outputs(
    y_out,
    target_offsets,
    target_lengths
)
```

shape：

$$
y_{out}^{tar} \in \mathbb{R}^{B \times T_{shift} \times D}
$$

### 8.2 v14.4c routing 输出 codebook-specific hidden

如果启用 codebook routing：

```python
routed_target_hidden = _apply_codebook_routing(...)
```

输出：

$$
H^{route} \in \mathbb{R}^{B \times Q \times T_{shift} \times D}
$$

如果 routing 没有输出，v14.5 会把同一个 hidden expand 到所有 codebook：

```python
codebook_head_inputs =
    y_out_target.unsqueeze(1).expand(B, Q, T, D)
```

因此进入 v14.5 hierarchical conditioner 前，总有：

$$
h \in \mathbb{R}^{B \times Q \times T_{shift} \times D}
$$

### 8.3 构造 shifted target tokens

训练时知道 ground-truth target tokens，因此可以构造 target segment 的 delayed token 序列：

```python
shifted_target_tokens, shifted_target_lens =
    processor.build_shifted_target_tokens(rearranged_y)
```

shape：

$$
Y^{shift} \in \mathbb{N}^{B \times Q \times T_{shift}}
$$

这里的 `Y_shift[b, q, p]` 表示：

```text
batch b
codebook q
delayed position p
```

上的 token id。

### 8.4 same-real-time earlier-codebook context

对于当前 codebook `q > 0`，遍历所有 earlier codebook：

```text
q_prev < q
```

同一真实时间的 earlier codebook 在 delayed sequence 中的位置为：

```python
prev_position = p - (q - q_prev)
```

代码逻辑：

```python
for q in range(1, n_codebooks):
    for q_prev in range(q):
        shift = q - q_prev
        prev_ids = shifted_token_ids[:, q_prev, :T-shift]
        prev_emb = audio_embedding(prev_ids, codebook_idx=q_prev)
        context[:, q, shift:, :] += prev_emb
        counts[:, q, shift:, :] += 1
```

公式写作：

$$
r_{p,q}
=
\frac{1}{q}
\sum_{q'=0}^{q-1}
E_{q'}(Y^{shift}_{p-(q-q'),q'})
$$

其中：

- $E_{q'}$ 是 codebook-specific audio embedding；
- 只有 valid 的 earlier token 才参与；
- 如果没有 earlier token，则不注入。

### 8.5 context projection

对 `q > 0`：

$$
\Delta h_{p,q}
=
\alpha_q \cdot \text{MLP}_q(\text{LN}(r_{p,q}))
$$

其中：

```text
α0 = 0.0
α1 = 0.03
α2 = 0.03
α3 = 0.03
```

代码对应：

```python
delta = context_projs[q](context[:, q, :, :])
delta = context_scales[q] * delta
```

### 8.6 hidden update

对 valid 位置：

$$
\tilde{h}_{p,q}
=
h_{p,q}

\Delta h_{p,q}
$$

对 invalid 位置：

$$
\tilde{h}_{p,q}
=
h_{p,q}
$$

代码：

```python
outputs[:, q, :, :] = torch.where(
    valid[:, q, :].unsqueeze(-1),
    outputs[:, q, :, :] + delta,
    outputs[:, q, :, :],
)
```

### 8.7 codebook head prediction

最终：

```python
logits = predict_layer(
    x=y_out_target,
    codebook_inputs=codebook_head_inputs
)
```

输出：

$$
\ell \in \mathbb{R}^{B \times Q \times T_{shift} \times V}
$$

其中 `codebook_inputs` 已经包含 v14.5 residual context。

然后通过 delayed pattern revert：

```python
aligned_logits, aligned_targets =
    processor.post_process_target_logits(logits, rearranged_y)
```

最后用原本 cross entropy 训练：

$$
\mathcal{L}_{CE}
=
\sum_{q=0}^{Q-1}
CE(\ell_q, c_q)
$$

## 9. v14.5 推理时完整数据流

训练时可以直接看到 ground-truth target tokens；推理时没有 ground truth，只能使用已经生成的历史 token。

### 9.1 AR step 循环

推理中每一步：

```python
for step in range(total_generate_len + n_codebooks + 2):
```

当前 decoder 输出：

$$
H_s
$$

代码：

```python
last_hidden = out[:, -1:, :]
```

### 9.2 routing / expand

如果有 v14.4c routing：

```python
routed_last_hidden = _apply_codebook_routing(...)
```

得到：

$$
h_s \in \mathbb{R}^{B \times Q \times 1 \times D}
$$

否则 expand：

```python
last_hidden.unsqueeze(1).expand(B, Q, 1, D)
```

### 9.3 从生成历史构造 same-real-time context

推理时，生成历史保存在：

```python
curr_generated[q]
```

对于当前 step `s`、当前 codebook `q`、earlier codebook `q_prev`：

```python
prev_step = step - (q - q_prev)
```

如果：

```text
prev_step >= 0
prev_step < len(curr_generated[q_prev])
```

说明同一真实时间的 earlier codebook 已经生成，可以取出：

```python
token = curr_generated[q_prev][prev_step]
```

再用 codebook-specific embedding：

```python
emb = audio_embedding(token, codebook_idx=q_prev)
```

聚合成 context。

### 9.4 residual context 注入

推理时同样：

$$
\tilde{h}_{s,q}
=
h_{s,q}

\alpha_q \cdot \text{MLP}_q(r_{s,q})
$$

然后送入 codebook heads：

```python
step_logits, step_reprs =
    predict_layer.forward_with_reprs(
        x=last_hidden,
        codebook_inputs=codebook_step_inputs,
    )
```

得到当前 step 四个 codebook 的 logits：

$$
\ell_{s,0}, \ell_{s,1}, \ell_{s,2}, \ell_{s,3}
$$

再采样：

```python
samples = sample_codebooks_token(...)
```

并追加到：

```python
curr_generated[q].append(samples[q])
```

这使下一步 residual codebook 可以使用当前已经生成的 token。

## 10. v14.5 的损失函数

`v14.5` 没有新增主要 loss。

仍然使用原本的 codebook CE loss：

$$
\mathcal{L}
=
\sum_{q=0}^{Q-1}
w_q
CE(\ell_q, y_q)
$$

其中当前主配置中：

```text
codebook_weights = [3.0, 1.0, 1.0, 1.0]
```

也就是说：

- q0 权重更高；
- q1/q2/q3 仍参与训练；
- v14.5 只是改变 codebook head 的输入 hidden，不额外加新监督。

## 11. v14.5 的训练细节

### 11.1 初始化

`v14.5` 基于 `v14.4c`：

```yaml
defaults:
  - /model/glo-var/v14.4c
```

训练脚本通常从 `v14.4c` checkpoint 初始化。

### 11.2 新增参数

只新增：

```text
hierarchical_codebook_conditioner
```

包括：

- `context_scales`
- `context_projs[q]` for q=1,2,3

q0 的 projection 是 `Identity`，但 scale 为 0，不注入。

### 11.3 为什么 scale 很小

初始：

```text
[0.0, 0.03, 0.03, 0.03]
```

这表示：

- 不动 q0；
- 只轻量修正 residual codebook；
- 避免一开始就破坏已有 decoder/head 分布。

这也是 `v14.5` 比 `v15.1a` 安全的核心原因。

## 12. v14.5 和 H/g/q 的关系

### 12.1 H 层负责什么

`H_s` 是 AR decoder hidden：

```text
H_s = decoder(text, ref audio, video/planner, generated history)
```

它负责全局语言、历史、视频条件整合。

### 12.2 g 层负责什么

每个 codebook head 会从 hidden 中产生中间表示：

```text
g_{s,q}
```

它是 codebook-specific 的预测空间。

### 12.3 q/token 层负责什么

q 层输出具体 codec token：

```text
c_{t,q}
```

### 12.4 v14.5 加在哪里

v14.5 不直接改最终 logits，也不直接改采样结果。

它加在：

```text
H_s -> codebook-specific hidden -> codebook head
```

之间。

更准确地说：

```text
base_hidden for each codebook
    + same-real-time earlier-codebook context
    -> codebook head
    -> g
    -> logits
```

所以它不是：

```text
直接改 q token
```

也不是：

```text
直接改 q0/g0
```

而是：

```text
给 residual codebook head 提供更合理的上下文
```

## 13. 为什么 v14.5 是 clean base

### 13.1 它的动机来自真实 codec 结构

EnCodec / RVQ 本来就是 residual quantization：

```text
q0: coarse acoustic representation
q1-q3: residual refinement
```

v14.5 利用这个结构，而不是额外发明复杂 planner state。

### 13.2 它不依赖难解释的辅助标签

不需要：

- MFA phone target；
- occurrence target；
- remaining target；
- stop target；
- viseme target；
- q0 controller target。

这让论文叙事更干净。

### 13.3 它不破坏主生成器

相比 `v15.1a`：

- 不直接训练 q0 adapter；
- 不强行让 planner state 修改 g0；
- 不在 q0 logits 上加 prior；
- 只轻量帮助 residual codebook。

### 13.4 它和 delayed-codebook 理论一致

我们已经反复确认 delayed pattern 下：

```text
当前 step 最核心的是 q0
q1/q2/q3 更像后续 residual 补充
```

v14.5 正好沿着这个结构建模。

---

## 14. v14.5 的局限性

### 14.1 它不能直接保证 LSE 改善

v14.5 改的是 codebook residual dependency，不是显式 sync objective。

因此它可能提升：

- acoustic consistency；
- residual codebook stability；
- WER tail；
- 语音质量。

但它不一定直接优化：

- LSE-D；
- LSE-C。

### 14.2 它没有 candidate selection 能力

v14.5 仍然是 generator。

如果同一个输入下多个采样候选质量差异很大，v14.5 自己不会知道哪个更同步。

这正是为什么后面需要：

```text
WavLM-AVSync reranking
```

### 14.3 它不是 empirical best 的唯一选择

已有实验中 `v14.6b epoch0` 经验指标更强。

但 `v14.6b` 的 planner state 设计更复杂，部分内容意义不稳定。

因此建议：

```text
v14.5 = clean base
v14.6b = strong empirical generator baseline
Rerank-2 = final sync-improvement module
```

---

## 15. 当前最终论文中的合理位置

### 15.1 作为 clean base generator

论文中可以把 `v14.5` 表述为：

> A delayed-codebook-aware hierarchical residual generator that explicitly models same-real-time residual codebook dependencies.

### 15.2 作为和 reranking 结合的生成器

完整最终系统：

```text
Text + reference audio + target video
    -> v14.5 generator
    -> K candidate audio
    -> WavLM-AVSync scorer
    -> selected candidate
```

### 15.3 和 v14.6b 的关系

如果最终 `v14.6b + rerank` 指标最好，可以写成 strong variant：

```text
PILOT-Dub strong generator variant
```

但理论主线建议仍然以 `v14.5` 为 clean base。

---

## 16. 一句话总结

`v14.5` 的核心不是“再加一个复杂 planner”，而是：

> 在 delayed codebook pattern 下，让 residual codebook `q1/q2/q3` 显式利用同一真实时间已经生成的 earlier codebook，从而把 RVQ 的 residual hierarchy 写入 codebook prediction 结构。

它适合作为论文 clean base，因为：

- 结构动机清楚；
- 不需要额外标签；
- 不直接破坏 q0；
- 和 delayed-codebook 理论一致；
- 比 v15 系列更容易解释和复现。

但它本身不是最终同步优化器，所以当前最合理的最终系统是：

```text
v14.5 clean generator
    +
WavLM-AVSync candidate reranking
```

同时报告：

```text
v14.6b epoch0 + WavLM-AVSync reranking
```

作为 empirical strongest variant。
